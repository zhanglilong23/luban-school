# 编排模式目录

鲁班学派支持的 Agent 编排模式及应避免的反模式。

> 参考：agent-skills orchestration-patterns.md（MIT License）

## 核心规则

**用户是唯一的编排者。角色不调用角色。** 鲁班不会调狄公，狄公不会调墨子。角色交接通过用户显式指令完成。

---

## 模式一：直接调用

单一角色，单一输入，单一输出。最省成本的模式。

```
用户 → @鲁班 执行契约 → 鲁班输出 [榫卯]
```

**适用**：任务清晰、边界明确、单角色可完成。

---

## 模式二：斜杠命令

将常用角色调用封装为斜杠命令，避免重复描述工作流。

```
/luban-school:luban → Skill("luban") → 按角色规程执行
```

**适用**：同一角色反复被调用，用户不想每次解释角色职责。

**反信号**：如果命令只是"决定调用哪个角色"，删掉它，让用户直接调用角色。

---

## 模式三：用户驱动顺序管线

用户按固定顺序运行角色，角色间通过交付物传递上下文。

```
用户: @问道 共修需求 → 问道输出需求真言 → 用户确认
用户: @司南 拆契约   → 司南输出契约 → 用户确认
用户: @鲁班 执行     → 鲁班输出 [榫卯] → 用户确认
用户: @狄公 审查     → 狄公输出放行单 → 用户确认
用户: @墨子 deploy   → 墨子部署完成
```

**适用**：有依赖关系的工作流，每步需要人工判断。

**为什么不让 Agent 自动编排**：LLM 编排者会 (a) 在交接时丢失信息 (b) 跳过人工检查点 (c) 加倍 token 成本。

---

## 模式四：七签叠加

将云笈七签的某一签叠加到目标角色，改变其行为深度而不改变职责边界。云笈是天书（功法总纲），不是角色——直接通过 `@角色 +签名` 修习。

```
用户: @鲁班 +入木 → 鲁班修习入木签，Debug 时深挖根因
用户: @狄公 +存真 → 狄公修习存真式，审查代码冗余
用户: @鲁班 +入木 +契约 → 鲁班多签叠加：Debug 同时严守契约边界
```

**适用**：需要临时改变角色的追问深度或思维模式，但不改变角色职责。

**与旧设计的区别**：旧设计中云笈是"原型切换器角色"，使用 `@云笈(入木) @鲁班` 两层间接调用；新设计中云笈降格为功法库，直接 `@鲁班 +入木` 一层修习。云笈不入轮回，留一篇《云笈七签》飘然而去。

---

## 模式五：研究隔离

当调研任务会污染主会话上下文时，派发 Explore 子 Agent。

```
主 Agent → Explore 子 Agent（读 50 个文件） → 摘要 → 主 Agent 继续
```

**适用**：跨模块代码搜索、依赖分析、大范围代码库探索。

---

## 反模式

### A. 路由角色

一个角色的唯一工作是决定调用哪个角色。

```
❌ /work → 路由角色 → "这个需要鲁班" → 调用鲁班 → 返回给路由 → 转述给用户
```

**为什么失败**：纯路由层无价值，增加两次转述损耗。用户本来就知道要调鲁班。

**正确做法**：用斜杠命令或 @mention 直接调用。

### B. 角色调用角色

鲁班内部调用狄公来审查自己的代码。

```
❌ 鲁班 → "让我叫狄公来审查" → 狄公 → 审查报告 → 鲁班
```

**为什么失败**：角色被设计为产生单一视角，串联破坏独立性。狄公必须对用户负责，不对鲁班负责。

**正确做法**：鲁班交付 [榫卯] 后，由用户决定是否调用狄公。

### C. 自动化管线编排者

一个 Agent 替用户跑完整个问道→司南→鲁班→狄公→墨子管线。

```
❌ 编排Agent → @问道 → @司南 → @鲁班 → @狄公 → @墨子（全程无人干预）
```

**为什么失败**：丢失所有人工检查点。需求真言确认、契约确认、放行单签发——这些是用户决策点，不能自动化。

**正确做法**：用户保持为编排者，每个角色调用由用户发起。

### D. 深层角色树

```
❌ /ship → 协调者 → 质量主管 → 鲁班（三层嵌套）
```

**为什么失败**：每层增加延迟和 token 消耗，叶节点角色丢失上下文。

**正确做法**：编排深度最多 1（用户 → 角色）。合并在用户侧完成。

### E. 复活云笈为角色

```
❌ @云笈(入木) @鲁班 → 把云笈当作"原型切换器角色"重新引入
```

**为什么失败**：云笈已在 v2 设计中降格为天书（功法总纲）。把它复活为角色 = 退回到两层间接调用的旧设计，且角色定位虚浮（"切换别人行为"不像角色，像开关）。

**正确做法**：直接 `@角色 +签名` 修习功法。云笈作为天书永远在线，不需要被 @召唤。

---

## 决策流

```
工作是否单一角色可完成？
├── 是 → 直接调用。结束。
└── 否 → 是否有依赖关系？
         ├── 是 → 用户驱动顺序管线。
         └── 否 → 子任务是否独立？
                  ├── 是 → 并行子Agent合并。
                  └── 否 → 回到顺序管线。
```

## 与 agent-skills 的对齐

鲁班学派的编排模式与 agent-skills 的 5 模式兼容，差异在于：

| agent-skills | 鲁班学派 |
|-------------|---------|
| Direct invocation | 直接调用 |
| Single-persona slash command | 斜杠命令 |
| User-driven sequential pipeline | 用户驱动顺序管线 |
| Multi-persona overlay | 七签叠加（`@角色 +签名`） |
| Research isolation | 研究隔离 |
