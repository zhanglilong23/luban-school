# 敕令·巡天仪 — 背景文档

## 起源

敕令·巡天仪的诞生源于一次真实的生产事故。

### 事件背景

2026年5月14日，敕令·周天仪在执行 TSK-A06（AgentChatService — SSE 流式输出聊天服务）任务时，子agent（鲁班）在验证阶段陷入死循环：

```
⚙️ 根因查清：鲁班 agent 已完成全部编码 + 编译通过，但在验证阶段挂住（可能陷入 MAF 源码确认循环）。法器接管续手。
```

**具体表现**：
- 鲁班在搜索 `ToolApprovalRequestContent` 类时陷入循环
- 执行时间超过21分钟，无任何进展
- 周天仪缺乏自检机制，无法检测到卡死状态
- 用户不得不手动干预，一天的产出几乎为零

### 问题根因

1. **周天仪缺乏自检机制**：没有超时检测、心跳监控，无法判断子agent是否卡死
2. **缺乏主动监控**：周天仪只被动等待子agent返回，不监控执行过程
3. **缺乏自动恢复**：检测到卡死后，没有自动恢复机制（重试、上下文切换、简化需求）

## 设计目标

### 核心目标

1. **主动监控**：周天仪具备自检能力，实时监控子agent执行状态
2. **自动恢复**：检测到卡死后，自动执行重试+上下文切换
3. **异步协作**：周天仪和巡天仪通过共享状态文件异步协作
4. **持续拆解**：巡天仪监控进度，通知司南拆解下一波TSK

### 设计原则

1. **不决策、不代劳、不评价**：法器只运转，不替三匠做决定
2. **闭环验证**：所有操作必须有证据，无证据不算完成
3. **事实驱动**：基于实际执行结果，不基于信心或猜测
4. **穷尽**：重试次数用尽再仪停，不提前放弃

## 架构设计

### 法器家族

```
敕令（法器总称）
  ├─ 周天仪（执行域）
  │   ├─ 调度鲁班、狄公、墨子
  │   ├─ 自检：超时检测、心跳监控
  │   └─ 巡查：执行进度监控
  │
  └─ 巡天仪（设计域）
      ├─ 监控代码走向（对比契约）
      ├─ 解决架构问题（仲裁偏离）
      └─ TSK持续拆解（动态扩展任务）
```

### 共享状态文件

**位置**: `docs/luban-school/shared/status.md`

**作用**：周天仪和巡天仪的异步通信桥梁

**信号类型**：
- 周天仪→巡天仪：heartbeat, stage_complete, stuck, deviation, complete
- 巡天仪→周天仪：continue, skip, abort, adjust
- 巡天仪→司南：need_new_tsk

### 卡死处理方案

**重试+上下文切换**（不是降级）：

| 重试次数 | 上下文切换 | 效果 |
|---------|-----------|------|
| 第1次 | 带上失败上下文 | 鲁班知道上次卡在哪里，避免重复 |
| 第2次 | 换一种实现方式 | 换思路，避免同样的死循环 |
| 第3次 | 简化需求 | 保证核心功能完成 |
| 第4次 | 终止，报告用户 | 请求人工决策 |

**简化需求的边界**：
1. 核心功能完整
2. 代码可编译
3. 基本流程可运行

### 偏离处理

| 偏离类型 | 判定条件 | 处理方式 |
|---------|---------|---------|
| `acceptable` | 版本升级(minor/patch)、工具链等价替换、实现细节调整但满足DONE | 记录，继续执行 |
| `architectural` | 设计方案变更（如REST→gRPC）、核心逻辑变更 | 写干预指令，等司南仲裁 |
| `requirement` | 实现了契约外功能、遗漏了契约内功能 | 写干预指令，等司南仲裁 |

## 使用方式

### 周天仪

```
/luban-school:chiling-zhoutian TASK-xxx
```

周天仪启动后：
1. 读取契约
2. 初始化共享状态文件
3. 启动周天自转（鲁班→狄公→墨子）
4. 每1分钟更新心跳
5. 检测卡死（超时>10分钟）
6. 卡死时执行重试+上下文切换

### 巡天仪

```
/luban-school:chiling-xuntian
```

巡天仪启动后：
1. 读取契约
2. 读取共享状态文件
3. 每3分钟巡检一次
4. 检查偏离信号
5. 检查卡死信号
6. 写入干预指令
7. 监控进度，通知司南拆解TSK

## 预期效果

### 解决的问题

1. **子agent卡死**：自检机制检测卡死，自动重试+上下文切换
2. **被动等待**：主动监控，实时掌握执行状态
3. **人工干预**：自动恢复，减少人工干预次数
4. **任务中断**：持续拆解，保证任务连续性

### 预期收益

1. **提高产出**：减少卡死导致的产出损失
2. **提高质量**：自动恢复机制保证核心功能完成
3. **提高效率**：异步协作，减少等待时间
4. **提高可维护性**：结构化日志，便于排查问题

## 未来演进

### 短期（v1.x）

1. 完善自检机制
2. 优化卡死处理方案
3. 完善偏离处理逻辑

### 中期（v2.x）

1. 增加预警机制（预测风险）
2. 增加复盘机制（总结经验教训）
3. 增加并行执行能力

### 长期（v3.x）

2. 自适应（根据任务特性调整参数）

## 总结

敕令·巡天仪是鲁班学派的一次重要升级，从被动等待转向主动监控，从人工干预转向自动恢复。它解决了子agent卡死的核心问题，提高了系统的可靠性和产出效率。

**核心价值**：
- 主动监控，实时掌握执行状态
- 自动恢复，减少人工干预
- 异步协作，提高执行效率
- 持续拆解，保证任务连续性
