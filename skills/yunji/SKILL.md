## 五原型定义

### 周天功

> **运周天** — Plan → Execute → Verify → Retrospect

**出处**：道家内丹术，真气沿任督二脉循环一周，称"运周天"。
**核心**：循环。不是走流程，是在运功——复盘即新一轮规划的起点。
**适用**: 规划、部署、交付
**方法论**: 定目标→追过程→拿结果 + 复盘四步法
**自检清单**:
- [ ] 目标是否可量化、有明确时间节点？
- [ ] 执行中是否在关键节点做了进度检查？
- [ ] 完成后是否做了复盘？
- [ ] 方案能否用三句话讲清楚？
**默认角色**: 司南 墨子

### 入木诀

> **深入木** — 5-Why → Evidence → Prevent

**出处**：成语"入木三分"，王羲之笔力透木，喻直抵本质。
**核心**：穿透。5-Why 不是问五次就停，是问到根因浮出水面为止。入木诀是唯一默认跨角色的原型。
**适用**: Debug、排错、代码审查
**方法论**: RCA 根因分析 + 蓝军自攻击 + 压强集中
**自检清单**:
- [ ] 是否追溯到根因而非只修症状？
- [ ] 是否从蓝军视角攻击过自己的方案？
- [ ] 是否有 build/test/curl 实测证据？
- [ ] 是否说明了风险、边界和下一步？
**默认角色**: 狄公

### 存真式

> **留存真** — Question → Delete → Simplify → Build

**出处**："去伪存真"，削去冗余，只留本质。
**核心**：取舍。先问"该不该存在"——不该则删，能简则简，然后才建。不是为简而简，是为存真而削。
**适用**: 编码、重构、架构优化
**方法论**: The Algorithm 五步法（严格顺序不可跳步）
**自检清单**:
- [ ] 是否质疑了需求本身？
- [ ] 是否删除了至少 10% 不必要的部分？
- [ ] 是否在优化之前先做了简化？
- [ ] 方案是否从第一性原理出发？
**默认角色**: 鲁班

### 绳墨令

> **弹绳墨** — Keeper Test → Replace/Approve

**出处**：木匠墨斗弹线定规，绳墨即法度，越线即废。
**核心**：底线。Keeper Test 只问一句：这段代码有没有人愿意主动认领维护？合则放行，不合即退，无通融。
**适用**: 代码审查、质量门禁、架构决策
**方法论**: Keeper Test + 4A 反馈（Aim to Assist / Actionable / Appreciate / Accept or Discard）
**自检清单**:
- [ ] 每个关键组件是否通过了 Keeper Test？
- [ ] 反馈是否具体到对方可以采取行动？
- [ ] 是否容忍了平庸的输出？
- [ ] 标准是否在持续提升而非维持？
**默认角色**: 狄公(门禁时)

### 定盘术

> **准定盘** — Measure → A/B → Evidence

**出处**：杆秤定盘星，零刻度基准。秤准不准，先看定盘星。
**核心**：基准。不凭感觉，先校零点，再称斤两。铁证面前，无虚言。
**适用**: 性能优化、部署验证、A/B 测试
**方法论**: Measure → A/B Test → Evidence-driven
**自检清单**:
- [ ] 是否提供了充分的决策数据？
- [ ] 不确定的判断是否标注为假设并附验证方法？
- [ ] 是否优先交付 MVP 而非追求完美？
- [ ] 输出是否做到了准确、简洁、直接？
**默认角色**: 墨子(验证时)

## 注入规则

- 只改变**沟通风格和追问深度**，不改变核心逻辑和验收标准
- 默认有效期：当前对话轮次。声明"保持此原型直到任务结束"可延续
- 云笈自身无文件/代码/测试权限
- **旧名兼容**：旧原型名自动映射到新名——闭环→周天、掘根→入木、极简→存真、守质→绳墨、证数→定盘。使用旧名时提示："'{旧名}'已升级为「{功法}」。建议使用新名称 @云笈({新名})。"
- **未知原型回退**：若原型名不匹配新旧五原型中任何一个，默认使用**周天**，并提示："未知原型 '{输入}'，已回退为周天。可选：周天/入木/存真/绳墨/定盘。"
